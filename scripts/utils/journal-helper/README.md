# Journal Helper

Infrastructure engineering suffers from a persistent friction problem: the gap between doing the work in a terminal and accurately documenting it later. Journal Helper is a custom Bash-based documentation pipeline built to solve this. It ensures that every architectural decision, operational change, and incident response is captured in real-time, treating documentation as an integrated part of the engineering workflow rather than a retroactive chore.

Journal Helper is a Bash-based documentation pipeline that eliminates the gap between doing terminal work and accurately capturing it. The system has five components: an entry router (`start.sh`), a template engine (`new-entry.sh`), a session wrapper (`session-start.sh`), a transcript parser (`session-parse.sh`), and an entry editor (`edit-entry.sh`).
The template engine performs dynamic entry-type discovery, conditional sequential ID generation, shell variable expansion into Markdown templates, and automatic index population for ADR entries. The session wrapper builds a scoped rc file that stubs startup-noise commands before sourcing the user's real environment, injects the `note()` phase-annotation function for the session duration only, and handles shell compatibility via `ZDOTDIR` redirect (`zsh`) or `--rcfile` (`bash`).
The transcript parser runs a sequenced cleaning pipeline: `perl` strips ANSI/VT escape sequences first while the ESC byte is intact, then `col -b` resolves backspace/overstrike sequences. This order is non-negotiable — reversing it causes `col` to consume the ESC byte, leaving orphaned sequence tails. The cleaned transcript is split on `note()` phase markers via `awk` and injected between HTML sentinels in the Markdown document, which are invisible in all renderers.

## Architecture

| Script | Role |
|---|---|
| `start.sh` | Entry point and workflow router |
| `new-entry.sh` | Template engine: discovery, ID generation, variable expansion, index population |
| `session-start.sh` | Session wrapper: rc scoping, blacklist stubs, shell compatibility, recording |
| `session-parse.sh` | Transcript cleaner and sentinel injector |
| `edit-entry.sh` | Entry browser and editor |

## Usage

Start the pipeline by running the entry script:

```bash
./start.sh
```

1. Select whether to create a new entry or edit an existing one.
2. Choose the entry type (e.g., Incident, ADR, Operations).
3. The script will open your $EDITOR to draft the initial context.
4. Close the editor. If the template has terminal tracking sentinels, the script willl drops you into a tracked terminal session.
5.  Do your work. Use the note command to label phases (e.g., note "Phase 1: Firewall rules").
6. Type exit. The transcript is cleaned, injected into your document, and the editor reopens for final review.

Documentation becomes easy when you can use the `note` command to track what you are doing in real-time, with minimal friction, and in-context. Your main task becomes editing, instead of reconstructing from memory.
